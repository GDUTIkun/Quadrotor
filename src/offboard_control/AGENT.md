# offboard_control 编码规范

本文件约束 `src/offboard_control` 内的 offboard 相关节点和脚本。这个包通常在主机电脑上编写、在树莓派上运行，所以修改时要优先保证可移植、可回滚、可现场调参。

## 基本原则

- 主要业务节点在 `src/*.cpp`，保持 ROS 2 / MAVROS 的现有写法，不为了小改动引入新的框架或复杂抽象。
- offboard 控制逻辑要显式表达状态机、目标点、容差和超时；不要把关键飞行参数写死在算法深处，优先使用 ROS 参数。
- 修改飞行流程、解锁、模式切换、降落、舵机/GPIO、RC 急停逻辑时，要把安全影响写清楚，并保留保守默认值。
- 日志使用 `RCLCPP_*`，高频循环中使用 throttle，避免树莓派运行时刷屏影响排查。
- C++ 代码保持当前风格：小函数、清晰命名、早返回；公共消息/服务 topic 名称不要随意改名。

## 主机与树莓派差异

- 主机环境可能只用于编译和编辑，不一定有 MAVROS、飞控、GPIO、舵机硬件。不要把硬件存在作为构造函数或编译期的强依赖。
- 树莓派上更在意稳定性和资源占用：避免无界线程、忙等循环、大量动态分配、持续文件写入和过高频日志。
- 涉及 GPIO/PWM/串口/设备路径的代码，要允许通过参数关闭或替换；默认行为不能导致硬件误动作。
- 不要提交只适用于个人主机的绝对路径、临时 IP、用户名、shell alias、ROS_DOMAIN_ID 等配置。

## 冲突处理

- 工作区可能同时被主机和树莓派改动。修改前先查看相关文件，不要覆盖自己没有写的改动。
- 遇到 CMake、launch、脚本安装项和实际文件不一致时，先确认目标是否仍在使用，再做最小修正。
- 不要运行破坏性 git 命令清理现场；生成文件、日志、bag、build/install/log 目录不要混入功能提交。

## 验证建议

- 能编译时至少运行：`colcon build --packages-select offboard_control`。
- 改动 offboard 状态机后，优先在仿真或断桨/安全台架环境验证，再上真机。
- 上树莓派前确认参数、topic、frame、模式名、RC 急停通道和降落流程；真机测试先用低高度、短航线、保守速度。
